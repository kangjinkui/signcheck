# [Claude Code] 맨땅에 헤딩 말고 오픈소스부터 — 비개발자 공무원의 AI 개발 도전기

## 📝 한줄 요약

업무를 맡은 지 2주 된 공공기관 담당자가 Claude Code와 함께 옥외광고물 허가/신고 자동 판정 시스템을 개발 중이다. 현재 판정 엔진 API가 완성됐고, 담당자마다 달랐던 판정 기준을 하나로 통일하는 것이 목표다.

**바쁘시면 이것만 읽어도 돼요:**
- Claude Code로 법령 기반 간판 허가/신고 자동 판정 API 구축 중 (현재 12종 광고물 커버)
- AI도 사람처럼 헷갈린다 — 법령명을 잘못 찾아오면 내가 직접 확인하고 수정해야 한다
- 핵심 팁: "RAG 챗봇, 오픈소스, 깃허브" 3개 키워드로 검색 → 내 요구사항에 맞는 걸 추천받기
- 버그를 AI와 함께 추적한 과정이 가장 인상적이었다 (원인이 DB가 비어있던 것)
- 맨땅에 헤딩하지 않아도 된다 — 이미 훌륭한 오픈소스가 있다
- AI가 잘 하고 있는지 항상 감시해야 한다

## 🎯 이런 분들께 도움돼요

## 😫 문제 상황 (Before)

업무를 맡은 지 2주째였다.

간판 허가 민원이 들어오면 담당자는 법령집을 꺼내 확인해야 한다. 돌출간판인지, 벽면간판인지, 몇 층인지, 면적이 얼마인지에 따라 허가 대상인지 신고 대상인지가 달라진다. 법령이 옥외광고물법, 시행령, 서울시 조례, 강남구 조례까지 4개 층위로 쌓여 있어서 처음 배우는 사람은 어디서부터 읽어야 할지부터 막힌다.

더 큰 문제는 **사람마다 판정이 다르다**는 것이다. 법령 해석에 따라 담당자가 "이건 허가예요"라고 했다가, 다른 담당자는 "신고면 돼요"라고 하는 일이 생긴다. 그러면 민원이 발생한다.

그래서 매일 옆 팀원에게 물어봐야 했다. 팀원도 바쁜데 계속 불러세우는 게 미안했다. _이 불편함을 없앨 수 없을까?_ 라고 생각하던 차에, CTO 오프 모임에서 힌트를 얻었다.

> "맨땅에 헤딩하지 마세요. 이미 훌륭한 개발자들이 오픈소스로 공유해둔 게 있어요."

그 말 한마디가 시작이었다.

## 🔧 작업 과정

### 오픈소스 검색부터 시작하다

CTO 모임에서 들은 조언을 바로 실행에 옮겼다. Claude Code에 딱 3개 키워드만 넣었다.

```
RAG 챗봇, 오픈소스, 깃허브
이 3개를 키워드로 내 요구사항에 맞게 오픈소스를 추천해달라
```

RAG(Retrieval-Augmented Generation)란 AI가 답변할 때 관련 문서를 먼저 찾아서 근거로 삼는 방식이다. 법령처럼 정확한 근거가 필요한 업무에 딱 맞는 기술이다. Claude Code는 LlamaIndex를 추천했고, 내 생각도 같았다. _나도 이걸 생각하고 있었는데._ 그냥 진행했다.

나중에 알게 됐지만, 이게 정말 중요한 방식이었다. 처음부터 만드는 게 아니라 이미 검증된 오픈소스를 내 상황에 맞게 조합하는 것. _왜 이걸 진작 안 했지?_ 라는 생각이 들었다.

---

### 법령 범위를 먼저 정했다

어떤 시스템을 만들 건지 Claude Code와 함께 구체화했다.

```
'/home/jinkui/dev/SignCheck/광고판정_PRD.md' 이거 분석해서 보고 해줘. 좀 더 구체화해보자
```

처음엔 간판 4종만 대상이었다. 그런데 법령을 보니 훨씬 많은 유형이 있었다.

```
4종으로 제한을 두지 말고 여기 파일에 있는 광고물은 다 대상이다
```

돌출간판, 벽면이용간판, 옥상간판, 지주이용간판, 입간판, 현수막, 애드벌룬, 선전탑, 창문이용광고물까지. 결국 12종 전체를 커버하게 됐다.

---

### 법령 API 연동 — AI도 헷갈렸다

국가법령정보공단 Open API를 연결해서 법령 데이터를 불러오는 단계였다. 그런데 여기서 흥미로운 일이 생겼다.

Claude Code가 서울시 조례를 잘못 찾아온 것이다. 내가 직접 검색해봐도 잘 안 나올 만큼 이름이 비슷한 조례가 여러 개 있었다.

_아, AI도 나처럼 헷갈리는구나._

사람이나 AI나 공식 명칭이 복잡하면 헷갈리는 건 마찬가지였다. 그래서 내가 직접 정확한 법령 정보를 찾아서 알려줬다.

```
서울시 조례는 이거 아니야.
서울특별시 옥외광고물 등의 관리와 옥외광고산업 진흥에 관한 조례(서울특별시조례)(제9936호)(20260105)
요거 찾아라
```

법령 ID를 직접 제공하자 바로 정확하게 찾았다. 이 경험으로 깨달은 것이 있다. **AI가 잘 하고 있는지 항상 감시해야 한다.** 특히 법령명이나 공식 코드처럼 정확도가 중요한 데이터는 내가 직접 확인하고 넣어줘야 한다.

---

### 버그 발견 — 그리고 함께 추적한 과정

법령 임베딩까지 마치고 Docker 환경에서 판정 API를 처음 실행했다. 돌출간판 허가 여부를 물어봤는데, 응답이 이상했다.

```
백엔드가 정상 시작됐습니다. 이제 /api/v1/judge 테스트합니다.
```

결과를 보니 최대 규격이 전부 `null`로 나왔다. 허가 판정도 `"report"`(신고)로 잘못 나왔다. 돌출간판은 분명히 허가 대상인데.

Claude Code와 함께 원인을 추적했다. 데이터베이스 테이블을 확인해보니 규칙 데이터가 아예 없었다. 0건. 시스템은 잘 돌아가는데 안에 데이터가 없으니 항상 기본값으로 빠지고 있었던 것이다.

이 과정이 가장 인상적이었다. 증상(null 반환)을 확인하고 → 원인(규칙 DB 비어있음)을 추적하고 → 해결책(규칙 데이터 삽입)을 찾는 과정을 AI와 함께 단계별로 밟아나간 것이다. 혼자였다면 어디서부터 봐야 할지 막막했을 텐데.

---

### 12종 간판, 15개 규칙 완성

버그 원인을 찾았으니 법령에서 규칙 데이터를 뽑아야 했다.

```
추가 테스트 필요하다. 돌출간판의 경우 가로 세로 최대 규격이 정해져 있다.
법규 목록에 모든 간판의 규격이 있다. 이건 매우 중요한 데이터다
```

서울시 조례 PDF를 분석해서 간판별 최대 규격을 추출하고, 15개 규칙을 데이터베이스에 넣었다. 돌출간판은 돌출폭 1미터 이내, 세로 3.5미터 이내. 벽면이용간판은 층수에 따라 허가와 신고가 나뉘고. 이런 복잡한 조건들이 하나씩 정리됐다.

최종 검증 결과:

```
돌출간판: 허가 | 소심의 | 돌출 1m 이내, 세로 3.5m 이하  ✓
벽면이용간판(5층): 허가 | 대심의 | 면적 225㎡ 이하       ✓
지주이용간판: 신고 | 소심의 | 높이 5m, 1면 3㎡ 이하      ✓
창문이용광고물: 신고 | 소심의 | 가로·세로 0.3m 이내      ✓
현수막: 신고 | 소심의 | 가로 0.7m, 세로 2m 이하          ✓
```

처음에 `"report"`(오류)로 나오던 돌출간판이 `"permit"`(정상)으로 바뀐 순간, 뭔가 해낸 것 같았다.

## 💬 이 과정에서 배운 AI 활용 팁

### 효과적이었던 것
1. **오픈소스 검색부터**: "RAG 챗봇, 오픈소스, 깃허브" 3개 키워드로 내 요구사항에 맞는 걸 추천받기. 처음부터 만들려 하지 말 것
2. **법령처럼 공식 자료가 있는 업무일수록 잘 맞는다**: 근거가 명확한 업무는 AI가 특히 잘 도와준다
3. **버그는 함께 추적한다**: 증상을 설명하면 원인까지 같이 파고든다

### 이렇게 하면 안 돼요
1. **AI 결과를 그냥 믿지 말기**: 법령명, 조례 번호 같은 공식 데이터는 반드시 직접 확인
2. **처음부터 다 만들려 하지 말기**: 이미 검증된 오픈소스를 조합하는 게 훨씬 빠르다
3. **감시를 게을리하지 말기**: AI가 잘 진행하고 있는지 매 단계 확인이 필요하다

## 🌍 다른 업무에 적용한다면?

이 방식은 **법령이나 규정을 근거로 판단해야 하는 업무**라면 어디든 적용 가능하다.

- 건축 허가 요건 사전 확인 시스템
- 식품 영업 허가 구비 서류 자동 안내
- 각종 민원 처리 기준 통합 조회 도구

공공기관이 아니더라도 사내 규정이나 계약 검토처럼 "기준이 복잡하고, 사람마다 판단이 달라서 민원이 생기는" 업무라면 비슷한 방식으로 자동화할 수 있다.

## 🚀 앞으로의 계획

지금은 API만 있다. 다음 단계는 **창구 직원이 실제로 쓸 수 있는 화면**을 만드는 것이다. 민원인이 간판 정보를 입력하면 바로 허가/신고 여부와 필요 서류가 나오는 형태로. 법령을 외우지 않아도 누구나 쓸 수 있는 도구가 목표다.

# [Claude Code] 계획은 대충 말고 끝까지 검증한다 - AI 개발 2탄

## 📝 한줄 요약

AI에게 일을 맡길 때는 "알아서 해줘"보다 "계획부터 검증해가며 해줘"가 훨씬 강력했다. 이번에 배운 핵심은 두 가지다. **계획은 완벽하려고 노력하게 만들고, 구현은 테스트로 계속 감시해야 한다.**

**바쁘시면 이것만 읽어도 돼요:**
- AI에게 바로 코딩시키기 전에 계획을 먼저 내놓게 하면 시행착오가 줄어든다
- 그 계획에 대해 "스스로 논리 오류가 없는지 검증해라"라고 한 번 더 시키면 품질이 올라간다
- 구현이 시작되면 테스트로 감시해야 한다. AI도 중간에 빗나갈 수 있기 때문이다
- 결국 중요한 건 "좋은 답변"이 아니라 "검증 가능한 작업 방식"이었다

## 🎯 이런 분들께 도움돼요
- AI에게 일을 맡겼는데 중간부터 산으로 가는 경험을 해본 분
- 비개발자지만 AI와 함께 업무 시스템을 만들고 있는 분
- 결과물보다 진행 과정의 신뢰도를 높이고 싶은 분

## 😵 이번에 부딪힌 문제

1탄에서 느낀 건 AI가 생각보다 빨리 많이 만든다는 점이었다. 그런데 그다음 문제가 생겼다.

**빨리 만드는 것과 제대로 만드는 것은 다르다.**

처음에는 "이 기능 만들어줘"라고 바로 요청했다. 그러면 Claude Code는 곧바로 파일을 고치고 코드를 작성했다. 속도는 빨랐지만, 가끔은 내가 원한 구조와 다르게 흘러갔다. 구현이 조금만 길어지면 중간에 전제가 흔들리거나, 한 부분을 고치면서 다른 부분의 일관성이 깨지기도 했다.

그래서 방식을 바꿨다. 코드를 바로 짜게 하지 않고, **먼저 계획을 세우게 했다.** 그리고 거기서 끝내지 않고 한 단계 더 넣었다.

> "네 계획을 스스로 검증해서 논리적 오류가 없는지 확인해라."

이 한마디가 생각보다 컸다.

## 🔧 작업 방식의 변화

### 1. 먼저 계획을 받는다

이제는 기능 요청을 하면 바로 구현부터 시키지 않는다. 먼저 이렇게 시킨다.

```text
구현 전에 작업 계획을 먼저 작성해라.
무엇을 바꾸는지, 왜 그렇게 하는지, 어떤 위험이 있는지 포함해라.
```

이렇게 하면 AI가 적어도 자기 머릿속 설계를 밖으로 꺼내놓는다. 내가 보기에도 편하고, 잘못된 방향이면 코드를 건드리기 전에 바로 수정할 수 있다.

예를 들어 판정 엔진 규칙을 손볼 때도, 어떤 조건 테이블을 수정할지, 어떤 API 응답이 바뀔지, 기존 데이터와 충돌은 없는지를 먼저 보게 된다. 이 단계만 있어도 불필요한 수정이 많이 줄었다.

---

### 2. 계획을 자기검증하게 한다

계획을 받는 것만으로는 부족했다. AI는 그럴듯한 계획도 잘 쓴다. 문제는 **그럴듯함과 타당함은 다르다**는 점이다.

그래서 이렇게 한 번 더 말한다.

```text
방금 작성한 계획을 스스로 다시 검토해라.
논리적 오류, 빠진 전제, 충돌 가능성이 없는지 자기검증해라.
```

그러면 AI가 자기 계획을 다시 보면서 빠진 부분을 채운다. 가령:

- 테스트 없이 리팩터링하려던 계획에 검증 단계가 추가되거나
- 데이터 구조를 바꾸면서 마이그레이션을 빼먹은 점을 스스로 발견하거나
- 프론트와 백엔드 응답 형식이 맞지 않는 문제를 미리 지적하기도 한다

사람으로 치면 "검토 한 번 더 해봐"에 가깝다. 별것 아닌 것 같지만, 이 과정이 들어가면 작업 정확도가 눈에 띄게 나아졌다.

## 👀 구현은 테스트로 감시한다

계획이 좋아도 구현이 항상 그대로 되지는 않는다. 특히 AI는 중간에 다른 파일을 건드리면서 처음 계획과 다르게 흘러갈 수 있다.

그래서 구현 단계에서는 **테스트를 감시 장치처럼 사용**하는 게 중요했다.

이번 프로젝트에서도 실제로 그런 순간이 있었다. 시스템은 정상 실행됐는데 결과가 이상했다. 최대 규격 값이 전부 `null`로 나오고, 허가 대상이 신고로 판정됐다. 이때 "코드는 돌아간다"는 사실은 아무 의미가 없었다. **테스트해보니 잘못 구현된 것이 드러난 것**이다.

결국 원인은 규칙 DB가 비어 있었기 때문이었다. 만약 테스트를 하지 않았다면, 겉으로는 동작하는 시스템으로 착각했을 가능성이 높다.

지금은 기능을 만들 때 이런 기준으로 본다.

- 계획이 맞는가?
- 구현 결과가 계획과 일치하는가?
- 테스트 결과가 기대값과 맞는가?

이 셋 중 하나라도 어긋나면 다시 봐야 한다.

## 💡 이번에 얻은 실전 팁

### 팁 1. "완벽한 계획"을 요구해도 된다

AI에게 너무 부담 주는 것 아닐까 생각할 필요가 없다. 오히려 처음부터 대충 시작하면 나중에 더 큰 비용이 든다.

이렇게 말하면 된다.

> 계획은 가능한 한 완벽하게 세워라.  
> 구현 전에 빠진 조건과 논리적 오류가 없는지 스스로 검토해라.

중요한 건 완벽을 100% 달성하라는 뜻이 아니라, **검증 없이 넘어가지 말라**는 뜻이다.

### 팁 2. 자기검증 문장을 꼭 넣는다

내가 가장 자주 쓰게 된 표현은 이것이다.

> 자기검증을 통해 논리적 오류가 없는지 확인해라.

짧지만 효과가 좋다. 계획 단계뿐 아니라 답변 정리, 규칙 해석, 리팩터링 제안에도 다 적용할 수 있다.

### 팁 3. 테스트는 선택이 아니라 감시다

테스트는 개발자가 안심하려고 하는 절차가 아니라, AI가 제대로 일하고 있는지 확인하는 감시 장치에 가깝다.

특히 아래 경우에는 꼭 필요했다.

- 법령 해석처럼 결과가 틀리면 민원으로 이어지는 업무
- 규칙 기반 판정처럼 조건 하나가 바뀌면 결과 전체가 달라지는 기능
- 리팩터링처럼 "겉보기엔 멀쩡한데 내부가 깨질 수 있는" 작업

## 📋 바로 써먹을 수 있는 프롬프트

### 프롬프트 1: 계획 먼저 제출시키기
> 구현 전에 작업 계획을 먼저 작성해라  
> 변경 대상, 이유, 예상 리스크, 검증 방법까지 포함해라

### 프롬프트 2: 자기검증 추가하기
> 방금 세운 계획을 스스로 다시 검토해라  
> 자기검증을 통해 논리적 오류, 빠진 전제, 충돌 가능성이 없는지 확인해라

### 프롬프트 3: 테스트로 감시하기
> 구현 후 반드시 테스트해라  
> 기대 결과와 실제 결과를 비교해서, 다르면 원인을 추적해라  
> 테스트 결과를 바탕으로 구현이 계획대로 되었는지 다시 판단해라

## 🔚 정리

이번에 가장 크게 배운 것은 AI를 잘 쓰는 사람은 말을 그럴듯하게 하는 사람이 아니라, **검증 절차를 설계하는 사람**이라는 점이다.

계획은 최대한 완성도 높게 세우게 하고, 그 계획을 스스로 다시 검토하게 하고, 마지막에는 테스트로 계속 감시한다. 이 흐름이 잡히고 나니 AI가 훨씬 믿을 만한 작업 파트너가 됐다.

## 🔧 작업 과정

### "동작한다"고 믿었던 게 사실은 절반짜리였다

하루를 시작하자마자 첫 번째 문제가 터졌다. 프론트엔드에서 추가 질문을 입력하면 오류가 뜨는 것이었다.

```
자 이제 구축된 프론트에서 실험을 하고 있는데, 추가질문을 했는데, 오류가 발생했습니다.
다시 시도해주세요라는 문구가 뜬다. 이거 고치고 싶다.
```

CORS 설정 문제였다. 고쳤다.

그런데 이걸 고치다 보니 더 큰 문제가 보였다.

```
db를 확인하고 싶다. 지금 문제가 예를 들어 돌출간판이 설치층수 제한과 면적 제한을
넘어도 "설치불가" 판정을 안해준다.
```

허가와 신고는 잘 됐는데, **"설치불가" 판정이 아예 없었다.** 층수를 아무리 올려도 시스템은 그냥 허가나 신고로 판정하고 있었다. 틀린 판정을 자신있게 내리고 있던 것이다.

1탄에서 "null 반환" 버그를 잡았던 것처럼, 이번엔 "아예 빠진 판정 유형"을 잡아야 했다.

---

### 하루 9개 세션 — 컨텍스트 관리가 관건이었다

세션이 길어지면 AI가 앞에서 한 말을 까먹는다. 오늘처럼 작업이 많은 날엔 특히 그렇다.

```
컨텍스트 관리를 위해 이번 세션은 종료하고 새로운 세션으로 작업하려고 하는데,
docs/sign-rule-expansion-plan.md 이거만 다음 세션에 제공해주면 이어서 작업하나?
아님 추가로 인수인계를 위한 handoff.md를 만드는 게 나을까?
```

Codex는 handoff.md를 만드는 게 낫다고 했다. 진행 상황, 완료 태스크, 다음 작업을 세션마다 기록해두는 방식이다.

세션이 끊겨도 handoff.md 하나만 넘겨주면 바로 이어받아서 진행했다. AI 도구가 바뀌어도, 세션이 끊겨도 — 맥락 문서만 살아있으면 작업은 계속됐다.

---

### 처음 설계의 핵심 기능을 걷어냈다

프로젝트 초기에 Ollama(로컬 LLM)를 이용한 RAG 시스템을 만들었다. 법령 조문을 벡터 DB에 넣고 판정 결과와 함께 근거 조문을 찾아주는 기능이었다.

그런데 실제로 써보니 너무 무거웠다. Ollama 하나가 메모리를 잔뜩 잡아먹었고, 배포도 복잡해졌다.

냉정하게 판단했다.

> 판정 엔진 자체는 DB 기반 규칙으로 돌아간다. Ollama가 없어도 허가/신고/불가 판정 결과는 달라지지 않는다.

Ollama를 걷어냈다. 근거 조문은 DB에서 직접 조회하는 방식으로 살렸다. 기능은 그대로인데 시스템이 훨씬 가벼워졌다.

_처음 설계를 버리는 게 아깝기도 했지만, 완성하지 못한 완벽한 시스템보다 실제로 동작하는 단순한 시스템이 낫다._

---

### 배포 — 비용 0원으로 해결했다

Ollama를 걷어내니 배포 선택지도 달라졌다.

처음엔 DigitalOcean VM을 검토했다.

```
그럼 나한테 디지털오션 VM이 있어. 거기에 올리는 형태로 할 수도 있겠네
4 vCPU / 8GB RAM 이 정도 하려면 요금제가 얼마지?
```

월 48달러였다. 예산 초과였다.

결국 선택지가 명확해졌다. 프론트엔드는 Vercel, 백엔드는 Render, DB는 Neon — 세 가지 모두 무료 플랜이 있었다. 환경변수로 연결하면 됐다.

서버 비용 0원으로 배포가 완성됐다.

---

### 배포 후에도 에러는 났다

배포를 완성하고 동료에게 링크를 보냈다. 써보니 판정 에러가 났다.

배포 환경이 달라지면서 생긴 문제였다. 다시 디버그했다. 그리고 결국 고쳤다.

_배포가 끝이 아니었다._ 하지만 그것도 결국 해결됐다.

## 💬 이 과정에서 배운 AI 활용 팁

### 효과적이었던 것
1. **설계가 무거우면 과감히 걷어낸다**: Ollama를 제거하면서 오히려 배포가 가능해졌다. 처음 설계가 전부는 아니다
2. **handoff.md로 맥락을 문서화한다**: AI 도구가 바뀌어도, 세션이 끊겨도 — 맥락 문서가 있으면 바로 이어진다
3. **무료 배포 조합은 충분히 실용적이다**: Vercel + Render + Neon 조합으로 실제 서비스 가능한 시스템을 0원에 올렸다

### 이렇게 하면 안 돼요
1. **배포했다고 끝이 아니다**: 환경이 달라지면 에러가 난다. 배포 후 직접 테스트는 필수
2. **처음 설계에 너무 집착하지 말 것**: "이미 만든 것을 버리기 아깝다"는 생각이 완성을 막는다

## 🌍 다른 업무에 적용한다면?

무료 배포 조합(Vercel + Render + Neon)은 이런 도구에도 그대로 쓸 수 있다:
- 사내 규정 자동 조회 도구
- 양식 작성 도우미
- 반복 문의 자동 응답 시스템

클라우드 비용이 없으니 부담 없이 시작할 수 있다.

## 🚀 앞으로의 계획

다음 주 팀 시연이 첫 관문이다. 실제 담당자들이 써보면서 나오는 피드백을 반영해서 규칙을 보완할 예정이다. 법령이 개정되면 규칙 DB만 업데이트하면 되도록 관리자 도구도 붙여둔 상태다.

# [Codex] 오픈소스 + 계획 + 자기검증 — 맨땅 헤딩 없이 AI 개발 2탄

## 📝 한줄 요약

하루 동안 AI와 9번의 대화를 이어가며 간판 판정 시스템을 완성했다. 핵심은 세 가지였다. **오픈소스를 활용하고, 계획을 세우고, 그 계획을 AI 스스로 검증하게 한다.** 맨땅에 헤딩하지 않아도 된다는 걸 몸으로 익힌 날이었다.

**바쁘시면 이것만 읽어도 돼요:**
- AI와의 대화가 길어지면 컨텍스트가 날아간다 — handoff.md 하나로 해결했다
- "TASK-A1 진행해" 처럼 잘게 끊어서 시키면 AI가 길을 잃지 않는다
- 노트북LM 같은 챗봇과 이 시스템의 차이: 판정 대상을 입력받아 **결정을 내리는** 시스템이다
- 돌출간판 6층 이상 "설치불가" 판정이 처음 나왔을 때 — 그 순간이 가장 인상적이었다
- AI가 엉뚱한 법령 조문을 붙여오기도 한다 — 결과를 항상 검토해야 한다
- 오픈소스 + 계획 + 자기검증 = 처음부터 만들 필요 없다

## 🎯 이런 분들께 도움돼요
- AI로 뭔가 만들어보고 싶은데 어디서부터 시작해야 할지 모르는 분
- AI에게 큰 작업을 맡겼다가 중간에 산으로 간 경험이 있는 분
- 단순 챗봇 말고 "실제로 판단해주는 시스템"이 뭔지 궁금한 분

## 😫 이번엔 어떤 문제였나

API가 완성됐다고 끝이 아니었다.

시스템이 돌아가는 건 맞는데, **판정이 틀렸다.** 돌출간판을 6층에 설치한다고 입력해도 "신고"로 나왔다. 법령상 명백히 "설치불가"여야 하는데. 설치 규격을 초과했을 때도 마찬가지였다. 시스템은 멀쩡히 동작하는데 결과가 틀리다 — 이게 생각보다 훨씬 위험한 상태다. 모르고 쓰면 민원이 생긴다.

그리고 또 다른 문제가 있었다. 작업이 길어지니까 **AI가 앞에서 뭘 했는지 잊어버리기 시작했다.** 새 대화창을 열면 처음부터 다시 설명해야 했다. 하루에 9번 대화를 새로 시작하면서, 매번 "현재 상황은 이렇고, 지금까지 한 건 이거고..." 를 반복하는 건 비효율이었다.

이 두 가지를 해결하는 게 이날의 목표였다.

## 🔧 작업 과정

### AI가 잊어버리기 전에 — handoff.md 도입

처음엔 그냥 새 대화를 열고 다시 설명했다. 그런데 세 번째 세션쯤 되니까 이게 너무 비효율적이었다. AI한테 같은 말을 반복하고 있는 나를 발견했다.

그래서 Codex에게 물었다.

```
컨텍스트 관리를 위해 이번 세션은 종료하고 새로운 세션으로 작업하려고 하는데,
sign-rule-expansion-plan.md 이거만 다음 세션에 제공해주면 이어서 작업하나?
아님 추가로 인수인계를 위한 handoff.md를 만드는 게 나을까?
```

Codex는 handoff.md를 추천했다. 단순히 계획서를 넘기는 것보다, **"지금까지 뭘 완료했고, 다음은 뭘 해야 하고, 어디서 막혔는지"** 를 정리한 문서가 훨씬 낫다고 했다. 그렇게 만들어졌다.

이후부터 세션을 끝낼 때마다 Codex가 handoff.md를 업데이트했다. 다음 세션을 열 때는 한 줄만 던지면 됐다.

```
docs/handoff-sign-rule-expansion.md 다음 작업 이어하자
```

그러면 Codex는 바로 파악하고 시작했다. 내가 설명할 필요가 없었다. _이게 이렇게 편할 줄이야._

---

### TASK 단위로 끊어서 시키기

판정 오류를 고치는 작업이 생각보다 범위가 넓었다. 돌출간판 하나만 고치면 되는 게 아니라, 12종 간판 전체의 판정 조건을 법령에 맞게 다시 검토해야 했다.

한 번에 다 맡기면 어떻게 될까? 써봤다. AI가 중간에 흔들렸다. 어디까지 했고 어디서 멈췄는지 스스로 헷갈려했다.

그래서 방식을 바꿨다. 먼저 전체 로드맵을 짜게 하고, 그걸 작은 태스크로 쪼갰다.

```
돌출간판에서 시작해서 모든 간판으로 판정을 확장하는 계획을 세우고 보고해줘
```

Codex가 TASK-A1부터 Epic D까지 구조를 짰다. 그다음부터는 이렇게만 말했다.

```
TASK-A1 진행해
```
```
TASK-A2 진행해
```

한 태스크가 끝나면 다음 태스크. 단순하지만 이게 훨씬 잘 됐다. AI가 지금 뭘 해야 하는지 명확하니까 길을 잃지 않았다. 그리고 내가 보기에도 진행 상황이 눈에 보였다.

---

### 그리고 마침내 — "좋네 잘 반영되네, 좋은데"

테스트를 해봤다. 돌출간판, 6층, 면적 초과. 판정 결과를 봤다.

**"설치불가."**

처음 보는 결과였다. 전날까지는 이 조건에서도 "신고"가 나왔다. 그게 드디어 "설치불가"로 바뀐 것이다.

```
좋네 잘 반영되네, 좋은데
```

짧은 한마디였지만, 그 순간이 가장 인상적이었다. 법령에서 규칙을 뽑고, 그걸 DB에 넣고, 엔진이 그 규칙대로 판정하고, 화면에 결과가 나오는 전체 흐름이 처음으로 완전하게 맞아떨어진 순간이었다.

---

### 막혔던 순간들

물론 매끄럽지만은 않았다.

**첫 번째 막힌 것**: 입간판을 11층, 30㎡로 입력했는데 "설치불가"가 안 나왔다.

```
아직 DB에 오류가 있다. 입간판의 경우 11층에 30제곱미터로 설치한다고 했는데
불가 판정이 안 나온다.
```

이건 DB에 입간판 층수 조건이 누락된 것이었다. 규칙은 법령에 있는데 DB에 안 들어가 있었다. 증상은 API 응답인데 원인은 데이터였다. 이 두 가지를 구분해서 볼 줄 알아야 찾을 수 있었다.

**두 번째 막힌 것**: 판정 결과와 함께 나오는 "근거 조문"이 전혀 엉뚱한 내용이었다.

```
근데 근거 조문은 전혀 상관없는 것이 생긴다.
```

RAG가 의미적으로 비슷한 문장을 가져오다 보니, 돌출간판 판정인데 현수막 조문이 붙어오는 일이 생겼다. 이건 완전히 해결하지 못한 채 넘어갔다. AI도 못 하는 게 있다. 그리고 그걸 인정하고 다음으로 넘어가는 것도 중요하다.

---

### 이 시스템이 노트북LM이랑 다른 이유

한 가지 더 깨달은 게 있었다.

누군가 "그거 그냥 노트북LM으로 법령 올려놓고 물어보면 되지 않아요?"라고 할 수 있다. 실제로 나도 처음엔 그 선택지를 고려했다.

차이는 이거다. 노트북LM은 **질문하면 답해준다.** 이 시스템은 **판정 대상을 입력받아서 결정을 내린다.**

민원인이 "돌출간판, 6층, 가로 2m, 세로 4m, 조명 있음"을 입력하면 시스템이 "설치불가 — 세로 3.5m 초과"를 반환한다. 담당자가 판단하는 게 아니다. 시스템이 법령 기준대로 판단한다. **담당자마다 달라지는 해석의 여지가 없다.**

챗봇과 결정 시스템의 차이. 그게 이 프로젝트의 핵심이었다.

## 💬 이 과정에서 배운 AI 활용 팁

### 효과적이었던 것

1. **handoff.md로 컨텍스트를 이어간다**: 새 세션을 열 때마다 AI에게 상황을 설명하지 말고, 직전 세션 AI가 남긴 인수인계 문서를 넘겨라. 그게 훨씬 빠르다.

2. **큰 작업은 TASK 단위로 쪼갠다**: "이 기능 다 만들어줘"보다 "TASK-1 진행해" → 완료 확인 → "TASK-2 진행해" 방식이 AI가 길을 잃지 않는다.

3. **계획에 자기검증을 붙인다**: 계획을 받고 나서 한 번 더 "이 계획에 논리적 오류나 빠진 부분이 없는지 스스로 검토해라"고 하면 품질이 올라간다.

4. **오픈소스를 먼저 찾는다**: 처음부터 만들려 하지 말 것. 이미 검증된 구조를 내 상황에 맞게 조합하는 게 훨씬 빠르고 안정적이다.

### 이렇게 하면 안 돼요

1. **AI 결과를 그냥 믿지 말기**: RAG 근거 조문이 엉뚱하게 나온 것처럼, AI도 틀린다. 특히 법령처럼 정확도가 중요한 데이터는 반드시 직접 확인.

2. **큰 작업을 한 번에 맡기지 말기**: 범위가 넓은 작업일수록 AI가 중간에 흔들린다. 잘게 쪼개서 하나씩 확인하면서 가야 한다.

3. **AI가 막혀도 억지로 밀어붙이지 말기**: RAG 조문 오류처럼 당장 해결이 안 되는 것도 있다. 인정하고 다음으로 넘어가는 것도 전략이다.

## 🌍 다른 업무에 적용한다면?

handoff.md + TASK 단위 방식은 간판 판정 시스템에만 쓰이는 게 아니다.

- 보고서 작성처럼 단계가 나뉘는 작업
- 데이터 정리처럼 반복적이지만 판단이 필요한 작업
- 여러 날에 걸쳐 이어가야 하는 프로젝트

어떤 업무든 "이 작업을 TASK로 쪼갤 수 있는가?" 를 먼저 물어보는 것. 그리고 세션이 끊길 때마다 AI에게 인수인계 문서를 남기게 하는 것. 이 두 가지만 해도 AI와의 협업 품질이 눈에 띄게 달라진다.

## 🚀 앞으로의 계획

이제 시스템은 완성됐다. 다음은 실제 창구에서 쓰이는 것을 보는 것이다.

강남구 내부망에 배포하고, 실제 담당자가 민원인 앞에서 판정 버튼을 누르는 장면. 그게 목표다. 코드가 아니라 **업무가 바뀌는 것**이 진짜 완성이다.

## 📋 재사용 가능한 프롬프트

### 프롬프트 1: 세션 인수인계 문서 만들기
> 이번 세션을 종료하기 전에 다음 세션을 위한 handoff 문서를 작성해줘
> 포함할 내용: 오늘 완료한 작업, 다음에 해야 할 작업, 현재 막힌 부분, 주의사항

### 프롬프트 2: 큰 작업을 태스크로 쪼개기
> [목표]를 달성하기 위한 전체 계획을 세워줘
> 독립적으로 실행 가능한 TASK 단위로 분리해서 목록으로 작성해줘
> 각 TASK는 완료 여부를 체크할 수 있게 명확하게 정의해줘

### 프롬프트 3: 계획 자기검증
> 방금 세운 계획을 스스로 다시 검토해라
> 논리적 오류, 빠진 전제, 단계 간 충돌 가능성이 없는지 자기검증해라
